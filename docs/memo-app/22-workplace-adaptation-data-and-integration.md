# 現場適応モード データ設計と既存統合

本ファイルは、現場適応モードのデータ保存方針の比較・推奨と、既存MindHubとの統合方針の正本とする。
ただし、確定したDBカラム・テーブルの正式定義は `03-data-model.md` が正本であり、DB変更は schema_version 方式に従う。
カテゴリ・visibilityの定義は `04-categories-and-tags.md` が正本。本ファイルではこれらを再定義せず参照する。

## 1. データ保存方針の選択肢

現場適応モードで扱うデータは、(a) 場面ごとのメモ（作業開始・詰まり・質問・報告・終業）と、(b) 現場プロファイル（現場ルール）である。保存方式には以下の3案がある。

### 1.1 案A：既存notes再利用（第一候補）

既存 `notes` テーブルをそのまま使う。

* 場面メモは `notes` の1レコードとして保存する（type / tags / visibility で場面・現場を表現）
* 現場プロファイルも `notes` の1レコード（またはtagで束ねる）として保存する
* 既存テーブルへの破壊的変更・カラム追加を行わない

### 1.2 案B：notesカラム追加

`notes` に現場適応モード用カラム（例：現場ID・場面種別）を追加する。

* schema_version 方式で追加のみ行う（既存カラムは変更しない）
* 場面・現場での絞り込みが構造化される

### 1.3 案C：専用テーブル新設

現場プロファイル・作業セッション用の専用テーブルを新設する。

* schema_version 方式で新テーブルを追加する（既存 `notes` は変更しない）
* 複数現場切り替え・セッション管理に強い

## 2. 各案の長所・短所

| 案 | 長所 | 短所 |
| --- | --- | --- |
| A：notes再利用 | 破壊的変更ゼロ、初期実装の負担が小さい、既存の検索・絞り込み・Markdown・アーカイブをそのまま使える | 場面・現場の構造化が弱く、tag/visibility運用に依存する |
| B：カラム追加 | 場面・現場の絞り込みが構造化される | schema変更が必要、fallback設計が要る、初期段階では過剰になりやすい |
| C：専用テーブル | 複数現場・セッション管理に最適 | 実装コスト最大、初期の仕様検証段階には重い |

## 3. 推奨方針

初期実装は **案A（既存notes再利用）を第一候補**とする。

理由。

* 既存 `notes` テーブルへの破壊的変更を避ける
* 初期実装の負担を抑える
* まずは仕様・導線・習慣化設計を検証する段階である

案C（専用テーブル）は、**現場プロファイルや作業セッション管理が既存notesでは不足すると判断された場合の将来候補**とする。案B（カラム追加）は、Aとの中間案として保持する。

いずれの場合も、正式なDB定義が必要になった時点で `03-data-model.md` を正本として更新し、schema_version 方式（`03` §6）に従う。既存 `notes` の破壊的変更は行わない。

## 4. 案Aでのデータ表現（初期実装）

案Aを採用する場合の、既存カラムでの表現方針（正式定義は `03`、カテゴリ値は `04`）。

* 場面の区別：`tags` に場面タグを付ける（`workplace_start` / `workplace_stuck` / `workplace_question` / `workplace_report` / `workplace_end` は**確定名ではなく例**。具体的な命名は未確定事項として `11-open-issues.md` §14 で管理する）
* 現場の区別：`tags` に現場タグ（現場名。ただし顧客名・非公開情報は入れない。`23`）を付ける
* `type`：既存type（worklog / thought 等）を再利用する。新type追加は次章の基準に従う
* `visibility`：既定 private（守秘優先の上書き。`23` §6.1）
* `is_git_candidate`：既定 false（守秘優先の上書き。`23` §6.1）
* `project`：現場を疑似プロジェクトとして扱ってもよい（既存の自由入力・候補選択を利用）

注意：既存type（worklog 等）は `04-categories-and-tags.md` §2 でGit候補初期値が `true` のものがあるが、現場適応モードではカテゴリ初期値に依らず既定を `is_git_candidate=false` / `visibility=private` に固定する（優先順位：現場適応モードの守秘ルール ＞ カテゴリ初期値。`23` §6.1）。

現場プロファイルは、専用の場面タグ（例：workplace_profile）を付けた1レコードとして保存し、表示画面（`21` 6章）で読み出す。

## 5. 現場プロファイルのデータ構造案

初期は単一現場で成立させ、複数現場切り替えは将来前提とする（`20` 7.2）。

保存候補項目。

* 現場名（一般化した呼称。顧客名は入れない）
* 使用ツール
* 質問先
* 報告先
* 報告形式
* ブランチ運用
* コミットルール
* 命名規則
* よく使う資料の場所（社内URL・非公開URLは入れない。場所の種別のみ）
* 保存禁止情報に関する注意

いずれも「職場情報の範囲」（`23` §3.1）に従い、職場固有の業務ルール・報告経路・判断基準・システム操作手順・例外対応・社内資料本文は保存せず、抽象化した個人用の覚書に留める。

案Aでは、上記をMarkdown本文またはキー・値の箇条書きとして1レコードに保存する。将来、複数現場化する場合は案B/Cへ移行し、`03` を更新する。

## 6. 既存仕様との統合

### 6.1 03-data-model.md との関係

* DBカラム・テーブルの正式定義は `03` が正本
* 案Aでは新規カラム・テーブルを追加しないため、初期は `03` へのスキーマ追記は発生しない（`03` §7に参照追記済み）
* 案B/Cへ移行する場合のみ、`03` に定義を追加し schema_version を更新する

### 6.2 04-categories-and-tags.md との関係

* type / visibility / tag の定義は `04` が正本
* 場面タグ・現場タグの命名を採用する場合、`04` のタグ運用に沿って記載する
* 新type追加は初期方針では行わず、既存type / tags / visibility の再利用を優先する。追加する場合は `04` を正本として定義し、20〜23では参照に留める

### 6.3 06 / 14 のプロンプト仕様との関係

* 質問文・報告文の生成テンプレート本文は `06`（整理プロンプト）・`14`（プロンプト集）が正本
* 現場適応モードは「どの場面でどのプロンプトを使うか」の対応を持つのみ（`21` 7章）

### 6.4 07-export-and-git-rules.md との関係

* 現場情報はGit候補・Markdown書き出し対象から除外する（既定 is_git_candidate=false）
* Git除外ルールの正本は `07`。現場適応モード固有の除外方針は `23` にまとめ、`07` へ参照追記する

## 7. 既存不変領域

* 既存メモ機能（`memos` テーブル／`app/memo/*`／GitHub手動アップロード）は変更しない
* 既存の主要導線・既存機能を壊さず、画面追加に留める（新規ルート追加は可。`21` 1章）
* 既存 `notes` テーブルへの破壊的変更は行わない

## 8. 出力物の扱い

* 初期方針はコピー中心（`23`、`20` 10章）
* notesへ保存する場合の既定：visibility=private、is_git_candidate=false、公開HTML・配布用・家族用に含めない
* 保存するかコピーのみかの最終判断は `11-open-issues.md` に未確定事項として残す
