# 現場適応モード 発表用サンプルデータ

現場適応モードを「見せられる・説明できる・意見をもらえる」状態にするための、発表・デモ用サンプル入力集。

* 対象画面：`/workplace` 以下の5場面（作業開始・詰まり記録・質問文作成・進捗報告作成・終業前メモ）
* このファイルはドキュメントのみ。実装コード・DB・app.json・package.json・ルート構造は変更しない
* サンプルはすべて架空の内容。実在の会社名・職場名・個人名・顧客名・秘密情報は含めない（守秘設計の正本は `23-workplace-adaptation-security-and-portfolio.md` §3.1）

## 1. 架空シナリオ

**未経験に近い新人が、社内向け「在庫管理Webアプリ」（架空）の画面確認タスクを担当する1日。**

* 登場するのは「自分（新人）」と「リーダー」のみ。固有名は使わない
* タスクは「会員登録画面の入力チェック確認（テスト仕様書の項目1〜20）」
* 5場面が1日の流れとしてつながる：朝の作業開始 → 昼前に詰まる → 質問する → 夕方に進捗報告 → 終業前メモ → （翌朝、再開導線のデモへ）

このシナリオを選んだ理由：画面確認タスクは未経験者が最初に任されやすく、専門知識がなくても聞き手がイメージしやすい。また「仕様書と実際の表示が違う」という詰まりは、新人が「不具合か、仕様書が古いのか、自分のミスか」を判断できず抱え込みやすい典型例のため、質問文作成の価値が伝わりやすい。

## 2. 5場面のサンプル入力

各表の「入力欄」はアプリの画面ラベルと一致させている。そのまま入力してデモに使う。

### 2.1 作業開始（/workplace/start）

| 入力欄 | サンプル入力 |
|---|---|
| 今日の作業 | 在庫管理Webアプリの会員登録画面の入力チェック確認。テスト仕様書の項目1〜20を順に確認する |
| 完了条件 | 20項目すべてを確認し、結果（OK / NG / 判断できず）を確認シートに記録してリーダーに報告する |
| 触る範囲 | 検証環境の会員登録画面のみ。データはテスト用アカウントだけを使い、他の画面や設定は変更しない |
| 先に確認すること（任意） | 検証環境のURLとテスト用アカウントの一覧の場所。NGを見つけたときの報告タイミング（都度か、まとめてか） |

**デモでの見せ方**：「今日やることを、完了条件と触ってよい範囲まで言葉にしてから始める」ことを見せる。新人は「どこまでやれば終わりか」「どこまで触っていいか」が曖昧なまま始めて事故を起こしやすい、という説明とセットにする。

**ChatGPTに貼ると**：出力を貼って「この作業計画に抜けがないか確認して」と頼むと、確認観点の漏れ（例：ブラウザの種類、確認結果の記録形式）を着手前に指摘してもらえる。

### 2.2 詰まり記録（/workplace/stuck）

| 入力欄 | サンプル入力 |
|---|---|
| 状況 | 項目12（メールアドレスの形式チェック）を確認中。誤った形式を入力したときのエラーメッセージが、テスト仕様書に書かれた文言と違っていた |
| 試したこと | 別のブラウザでも再現することを確認した。テスト仕様書の版数と更新日を確認した。他の項目（1〜11）では文言の食い違いはなかった |
| 確認したいこと | 仕様書が古いのか、アプリ側の不具合なのか。どちらとして記録すればよいか |
| エラー内容（任意） | 仕様書では「メールアドレスの形式が正しくありません」だが、実際は「入力内容を確認してください」と表示される |

**デモでの見せ方**：「詰まったらまず状況を言語化する。この出力がそのまま次の『質問文作成』の下書きになる」という流れを見せる。

**ChatGPTに貼ると**：出力を貼って「この詰まりの切り分け方を教えて」と頼むと、自分か仕様書かアプリかの切り分け手順を整理してもらえる。人に聞く前の自己解決の試行にも、聞く内容の整理にも使える。

### 2.3 質問文作成（/workplace/question）

| 入力欄 | サンプル入力 |
|---|---|
| 聞きたいこと | 項目12のエラーメッセージが仕様書と実際で違う場合、どちらを正としてNG判定すればよいか |
| 背景 | 会員登録画面の入力チェック確認（項目1〜20）を担当中。項目12のメールアドレス形式チェックで、仕様書の文言と実際の表示が一致しない |
| 自分で確認したこと | 仕様書の版数が最新であることを確認した。項目1〜11では文言の食い違いはなかった |
| 試したこと | 別ブラウザで再現を確認した。入力値を変えても同じ文言が表示された |
| 相手に判断してほしいこと | 仕様書とアプリのどちらを正とするか。NGとして記録するか、仕様書の修正依頼として扱うか |
| 急ぎ度（任意） | 今日中（この項目だけ保留にして先の項目に進んでいます） |

**デモでの見せ方**：詰まり記録の内容が「相手が答えやすい質問」に変わることを見せる。出力の冒頭に結論が来るため、「丸投げに見えない・調べた形跡が伝わる・相手は判断だけすればよい」という点を強調する。

**ChatGPTに貼ると**：チャットで人に送る前に「この質問文をもっと簡潔にして」と磨ける。また、そのままChatGPTに質問として貼れば、リーダーの回答を待つ間に一般的な考え方（テスト仕様書と実装の不一致の扱い方）を先に学べる。

### 2.4 進捗報告作成（/workplace/report）

| 入力欄 | サンプル入力 |
|---|---|
| 現在の状態 | 予定どおり進行中。20項目中17項目の確認が完了、1項目が回答待ち |
| 今日やったこと | 会員登録画面の入力チェック確認（項目1〜17。項目12は保留にして先へ進めた） |
| 完了したこと | 項目1〜11、13〜17の確認と結果記録（すべてOK） |
| 残っていること | 項目18〜20の確認。項目12の再確認 |
| 詰まっていること（任意） | 項目12はエラーメッセージの文言が仕様書と異なり、どちらを正とするか回答待ち |
| 次にやること | 明日午前に項目18〜20を確認し、項目12の回答が来しだい再確認して全項目の結果をまとめる |
| 相談・確認したいこと（任意） | 確認結果シートの提出先は共有フォルダでよいか |

**デモでの見せ方**：「結論（現在の状態）が先頭、相談は最後」という報告の型がアプリ側に組み込まれているため、報告が苦手な新人でも埋めるだけで型どおりになることを見せる。

**ChatGPTに貼ると**：出力を貼って「日報向けに3行に要約して」「チャット向けの丁寧な文体にして」と頼むと、提出先ごとの文体調整ができる。報告の中身を考える作業と、文章を整える作業を分離できる。

### 2.5 終業前メモ（/workplace/end）

| 入力欄 | サンプル入力 |
|---|---|
| 今日やったこと | 会員登録画面の入力チェック確認。項目1〜17まで完了（項目12のみ保留） |
| 未完了 | 項目18〜20の確認。項目12はエラーメッセージの文言の件で回答待ち |
| 明日最初にやること | 項目12の回答を確認してから、項目18〜20の確認に着手する |
| 再開のための補足（任意） | 検証環境のテストデータは毎朝リセットされるため、テスト用アカウントの再作成から始めること |

**デモでの見せ方**：この場面だけ「再開メモとして保存」を実行する。保存すると `/workplace` 入口の上部に「前回の再開メモ」として表示され、「この続きから作業開始」を押すと作業開始フォームに引き継がれる（明日最初にやること→今日の作業、未完了・補足→先に確認すること）。**5場面のデモの締めから翌朝の再開デモへつながる**ため、発表の流れが作りやすい。

**ChatGPTに貼ると**：翌朝、再開メモを貼って「今日の段取りを提案して」と頼むと、前日の状態を踏まえた1日の計画に展開できる。

## 3. 発表用デモの推奨手順

1. **事前準備**：デモ前日に（または直前に）§2.5の終業前メモを入力して保存しておく。入口に「前回の再開メモ」が表示された状態でデモを始められる。また、§2のサンプル文は事前にテキストファイルへ用意しておき、デモ中はライブで手入力せずコピーして貼り付ける（入力時間の短縮とタイプミス防止のため）
2. `/workplace` 入口を表示：「開始・作業中・終了」の3区分と5場面カードを見せ、毎日の仕事を一日の流れに沿って整理する導線であることを説明する。各カードの「コピーのみ／コピーと保存」ラベルと、開始区分内の「前回の再開メモ」から作業開始へつながる配置も示す
3. 作業開始 → 詰まり記録 → 質問文作成 の順に§2.1〜2.3を入力し、「整理する」→出力→「コピー」を見せる（この3つで「詰まっても抱え込まない」流れが伝わる）
4. 進捗報告作成（§2.4）で「結論が先頭に来る報告の型」を見せる
5. 終業前メモ（§2.5）を保存し、入口に戻って「この続きから作業開始」→引き継ぎ初期入力を見せて締める
6. 守秘の説明：入口と各入力画面に常時表示される注意文を示し、実名・顧客情報・認証情報・社内URLなどを入力せず、内容を一般化した個人用の覚書に留める設計であることを説明する。保存されるのは終業前メモのみで、private・Git候補外に固定される（`23` §3.1・§6.1）

補足：時間が短い場合は 3 を「詰まり記録→質問文作成」の2画面に絞ると、このモードの中心価値（抱え込まずに聞ける形にする）だけを最短で見せられる。

## 4. サンプル作成時の守秘ルール（このファイルの前提）

* 会社名・現場名・個人名は入れない（「リーダー」「自分」などの役割名のみ）
* 対象アプリは架空の「在庫管理Webアプリ」。実在のサービス・社内システムを連想させる固有名は使わない
* URL・アカウント名・システム名の具体値は書かない（「検証環境のURL」「テスト用アカウント」のような一般名にとどめる）
* サンプルを差し替える場合もこのルールに従う
