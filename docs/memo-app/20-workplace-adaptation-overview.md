# 現場適応モード 全体概要

本ファイルは「現場適応モード」の方針・目的・スコープの正本とする。
機能導線は `21-workplace-adaptation-flows-and-ui.md`、DB・統合方針は `22-workplace-adaptation-data-and-integration.md`、守秘・ポートフォリオ説明は `23-workplace-adaptation-security-and-portfolio.md` を参照する。

## 1. 背景

### 1.1 対象が直面する課題

IT未経験者・訓練校生・SES配属直後の人は、新しい開発現場・配属先・プロジェクトに入った直後に、以下で毎日つまずく。

* 今日何をやるべきか、どこまでやれば完了かが曖昧なまま作業を始めてしまう
* 詰まったときに、状況を整理できず抱え込む
* 質問したいが、何をどう聞けばよいか組み立てられない
* 進捗報告を求められても、報告文の形にできない
* 終業時に、どこまでやったか・明日何から再開するかを残せない
* 現場ごとのルール（質問先・報告形式・ブランチ運用・命名規則）が覚えきれない
* 何を保存してよく、何を保存してはいけないか（守秘）が分からない

ここでの「現場」は、物理的な作業現場ではなく、**IT開発の配属先・開発現場・SES現場**を指す。

### 1.2 既存機能だけでは習慣化しない理由

MindHub_Appには既にメモ管理（notes）・整理プロンプト・プロンプト集がある。
しかし、これらは「必要なときに自分で開いて使う」前提のため、未経験者が現場適応の初期に自力で使いこなすのは難しい。用語辞典やメモ帳を「見るだけ」では習慣化しない。

## 2. 目的

### 2.1 必ず発生する場面に導線を組み込む

現場適応モードの目的は、実務で毎日必ず発生する場面（作業開始・詰まり・質問・報告・終業）に、アプリの導線としてメモ・整理・出力を組み込むことである。

### 2.2 使った直後に出力が得られる

完璧な記録を残すことではなく、現場で迷わず動くための最低限の記録と、使った直後に得られる出力（質問文・報告文・再開メモ）を重視する。

### 2.3 質問へ切り替える目安を補助する（2026-07-23 Phase 16B 追記）

詰まりの言語化（詰まり記録）だけでなく、**まだ調べるべきか・質問へ切り替えるべきか**の目安を補助する。

* 目的は正解の自動決定ではなく、質問タイミングを安全側に整理し、質問文作成へつなげること
* 職場で質問・確認のルールが定められている場合は、そのルールを優先する
* 危険・高影響条件（本番・顧客・個人情報・認証・権限・削除・セキュリティ等）は時間を待たず確認する
* 詳細（判定条件・結果・受入条件）は `32-workplace-question-timing-rules.md` を正本とする

これは5場面モデルへの場面追加ではなく、**詰まり→質問**の補助導線である（5場面モデル自体は §7 のまま変更しない）。

## 3. 対象ユーザーと利用場面

* IT未経験者・訓練校生・SES配属直後の開発者
* 主な利用場面はPC作業中（初期スコープはPC用Webアプリ。`8. 対象プラットフォーム` 参照）

## 4. 現場適応モードの定義

現場適応モードは、MindHub_Appの**追加モード**である。

* 新しい開発現場・配属先に早く適応するための作業支援モードである
* 本体の通常メモ（notes）・整理プロンプト・カテゴリ・テンプレート・プロンプト集と連携する
* 本体機能を置き換えたり、既存機能を停止したりしない
* 物理的な屋外・作業現場のためのモードではない

## 5. 解決したい課題（機能の柱）

現場適応モードは、以下を導線として提供する。詳細な場面設計は `21` を参照。

* 作業開始時に、今日の作業・完了条件・触る範囲・確認事項を整理する
* 作業中に詰まったとき、状況・試したこと・確認したいことを整理する
* 質問前に、質問文を作成できるようにする
* 報告前に、進捗報告文を作成できるようにする
* 終業前に、未完了事項・明日最初にやること・再開メモを残す
* 現場ごとのルール・質問先・報告形式・ブランチ運用・命名規則を保存できるようにする（現場プロファイル。`22` 参照）
* 守秘情報を保存しないための注意・チェックを導線に入れる（`23` 参照）

なお、ここでの「職場情報」は個人情報・顧客情報だけでなく、職場固有の業務ルール・報告経路・判断基準・システム操作手順・例外対応・社内資料の内容も含む。現場適応モードは職場固有のルールや内部手順を蓄積する機能ではなく、抽象化された個人用チェック項目に留める（範囲の正本は `23` §3.1）。

## 6. 採用者・現場担当者の不安材料と回答方針

本モードはポートフォリオとして、複数現場を経験した採用担当者・現場担当者に見せる可能性がある。
単なる便利機能の羅列ではなく、以下の不安に答えられる構成にする。

| 不安材料 | 回答方針 | 詳細 |
| --- | --- | --- |
| 守秘情報を保存してしまうのでは | 保存禁止情報の明示、既定値private、AI貼り付け前の注意、公開・配布物への非混入 | `23` |
| 機能を盛りすぎて実務で使わないのでは | 毎日必ず発生する5場面に絞り、MVPは3場面優先 | 本ファイル7章、`21` |
| 既存アプリと整合していないのでは | 既存notes再利用を第一候補にし、正本（03/04/07/08/17）へ参照で接続 | `22` |
| 実務運用に非現実的なのでは | 入力は短く選択式・テンプレート式優先、コピー中心 | `21`、`22` |

## 7. スコープ

### 7.1 初期実装で対象にすること

* 5場面モデル（作業開始／詰まり記録／質問文作成／進捗報告作成／終業前メモ）の導線設計
* 現場プロファイル（単一現場で成立する範囲）
* 生成テキスト（質問文・報告文・再開メモ）のコピー（※Phase 16Cで、コピーを残したうえで `ChatGPTなどへ共有`（OS／ブラウザ共有）を追加する。詳細は `33`）
* 守秘チェックの導線への組み込み

MVPの優先順位は以下とする。

優先実装（毎日の習慣化と抱え込み防止に直結）。

1. 作業開始
2. 詰まり記録
3. 終業前メモ・翌朝再開

次点（作業ログが蓄積されてから価値が高くなる）。

4. 質問文作成
5. 進捗報告作成

### 7.2 非スコープ（初期実装で行わないこと）

* 用語辞典化（見るだけの知識データベース化はしない）
* 完璧な作業記録の強制
* 現場情報の自動同期・クラウド保存
* 現場プロファイルの複数現場切り替えの実装（設計は将来前提。初期は単一現場で成立させる。`22` 参照）
* オフライン機能の新規拡張・端末間同期（将来候補。`8` 参照）※「Android APK版での利用」自体は将来候補ではなく、Phase 16では実装・検証対象（`8`）。オフライン拡張と同一視しない
* 質問文・報告文のAI API自動生成（コピー運用を基本とする）
* 開発リファレンス（手順・コマンド・最小コード例の参照集）の機能化。開発リファレンスは現場適応モードとは**別機能**であり、本モードの中核ではない。5場面で手が止まったときに横から参照する補助導線として扱う。方針の正本は `24-development-reference.md`。初期は既存notesの `command` カテゴリ／タグ運用で扱い、専用DB・専用画面は作らない

## 8. 対象プラットフォーム

* Phase 14当初はPC用Webアプリ（`npx expo start --web`）を優先し、Android APK版・オフライン利用は将来候補としていた（歴史的経緯として残す）
* **現行のPhase 16では、Android APK版も実装・検証対象とする**。Phase 16B（質問タイミング画面）・Phase 16C（OS共有）はAndroid実機確認を受入条件に含む（`32`・`33`。Android運用の正本は `16-platform-and-distribution.md`）。具体的にはAndroidで質問タイミング画面とOS共有画面を確認する
* 一方、**オフライン機能の新規拡張・端末間同期は引き続き将来候補**とする（`16` §4）。「Android利用」と「オフライン拡張」を同じ将来候補として扱わない
* 将来的にスマホで現場ルールや再開メモを確認できる可能性は残す（`16-platform-and-distribution.md`、`13-mobile-view-export.md` の閲覧手段と整合させる）

## 9. MindHub本体との境界

* 既存メモ機能（`memos` テーブル／`app/memo/*`／GitHub手動アップロード）は変更しない
* 既存 `notes` テーブルへの破壊的変更は行わない（`03-data-model.md` の schema_version 方式に従う。`22` 参照）
* 現場適応モードは notes・整理プロンプト（`06`）・プロンプト集（`14`）・カテゴリ／テンプレートと連携する
* DB定義の正本は `03`、カテゴリ・visibilityの正本は `04`、画面フロー横断正本は `08`、Git除外ルールの正本は `07`、配布区分の正本は `17`。20〜23はこれらを再定義せず参照する

## 10. MVP範囲まとめ

* 正本として扱うのは5場面モデル（`21`）
* 初期実装は 作業開始／詰まり記録／終業前メモ・翌朝再開 を優先
* 出力はコピー中心。保存する場合の既定は visibility=private・Git候補false（`23`）
* DBは既存notes再利用が第一候補（`22`）
