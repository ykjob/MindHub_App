# 開発リファレンス 方針

本ファイルは「開発リファレンス」機能の方針・目的・スコープの正本とする。
開発リファレンスは現場適応モード（`20-workplace-adaptation-overview.md`〜`23`）とは**別機能**であり、現場適応モードの中核機能ではない。既存notesの `command` カテゴリ／タグ運用の上に成り立つ補助参照の位置づけとして扱う。

## 1. 目的（2026-07-12 改定：2本柱）

開発リファレンスは、単語を覚えるための辞書ではない。以下の2本柱で構成する（2026-07-12 改定。改定前は柱1のみを目的としていた）。

### 柱1：技術リファレンス

作業中に迷ったときに、手順・コマンド・最小コード例をすぐ確認し、集中力を切らさずに作業へ戻るための参照集。

* 目的は「暗記」ではなく「即座に見返して手を止めない」こと
* 作業中に手が止まった瞬間、横から短時間で確認して作業へ復帰できることを最優先にする
* 事故駆動で貯める（実際に手が止まったものを1件ずつ登録。§5.1）。R-01〜R-31（`25` §3〜§5）が該当

### 柱2：現場理解カード

実際の開発現場でどう動くか——現場に入って最初に押さえるべきこと、作業の各場面での確認手順、実務上の判断基準——を短く参照できる実務理解カード。現場に入った時に困らないための備えとして使う。

* 単なるコマンド集ではなく「確認手順・判断基準」として書く（何を・どの順で確認し、何を基準に判断するか）
* 行動基準なので事故駆動に限らず、初期セットとしてまとめて登録してよい（§5.2）。G-01〜G-20（`25` §10、登録用本文は `27-development-reference-fieldwork-cards.md`）が該当
* ポートフォリオ上「仕事に対する姿勢」を説明できる素材を兼ねる

### 両柱共通の制約

* 網羅性・体系性・学習教材化は目的にしない（それは公式ドキュメント・書籍の役割）
* 現場固有情報・機密は登録しない（§4）

## 2. 現場適応モードとの関係

* 現場適応モードは、作業開始・詰まり記録・質問文作成・進捗報告作成・終業前メモの**5場面**を支援する（`20` §5・§7.1、`21` §2）
* 開発リファレンスは、それらの場面で手が止まったときに**横から参照する補助導線**として扱う
* 開発リファレンスは現場適応モードの5場面には数えない。5場面モデル（`21` §2）や翌朝再開導線（`21` §4）を変更・拡張するものではない
* 位置づけの優先順位：現場適応モードの5場面支援が主、開発リファレンスは詰まったときの参照補助
* 守秘の考え方は現場適応モードと同じ方向（現場固有情報・機密は登録しない。§4・§5、`23` §3.1）だが、開発リファレンスは現場適応モード専用の守秘既定強制（`visibility=private` / `is_git_candidate=false` の上書き。`23` §6.1）とは独立して、既存notesの `command` カテゴリ運用に従う

## 3. 対象にするもの

柱1（技術リファレンス）：作業中に見返して、コピペまたは短い確認で作業へ戻れるもの。

* よく使う確認手順
* よく使うコマンド
* 関数・メソッドの最小例
* ライブラリの最小利用例
* エラー時の確認順
* 会話でよく出る一般IT用語の短い説明

柱2（現場理解カード）：現場での動き方の確認手順・判断基準。

* 現場参加・タスク着手前の確認事項（初日確認・タスク受領時確認・仕様書の見方・コードの読み順・影響範囲確認）
* 質問・報告・レビュー依頼の整理手順（質問前チェック・質問タイミング・報告内容・レビュー依頼前確認）
* トラブル対応の手順（エラー切り分け順・再現手順の書き方・ログの見方）
* Git・DB・テストの確認手順
* 守秘の判断基準（守秘情報を扱わない基準・現場固有情報をメモに残す時の注意）
* 業務の区切りの手順（終業前の再開情報・翌朝の再開手順）

## 4. 対象にしないもの

以下は開発リファレンスの対象にしない。

* AIが重要と言っただけの用語（実際にまだ使っていないもの）
* まだ実際に使っていない知識・先回りの知識（柱1に適用。柱2の現場理解カードは行動基準のため「まだ経験していない場面」の備えを含んでよいが、その場合も具体的な確認手順・判断基準として書けるものに限る）
* 主要ライブラリの網羅的な関数一覧
* 公式ドキュメントの丸写し
* 長文解説・体系的な教材
* 現場固有のシステム名・顧客名・内部URL・業務手順（`23` §3.1 の職場情報の範囲に該当）
* 機密情報を含むコマンドや設定値（認証情報・接続文字列・内部エンドポイント等）

## 5. 登録基準

### 5.1 柱1（技術リファレンス）の登録基準

以下のいずれかに当てはまるものだけを登録する。基準を満たさないものは登録しない。

* 実際に作業中に手が止まったもの
* 2回以上調べたもの
* 1分以内に見返したいもの
* コピペまたは短い確認で作業へ戻れるもの

### 5.2 柱2（現場理解カード）の登録基準（2026-07-12 追加）

現場理解カードは行動基準のため事故駆動を条件にしない。以下をすべて満たすものだけを登録する。

* 実務の特定の場面（着手前・質問・報告・トラブル・Git/DB・区切り）に紐づいていること
* 「いつ見るか」「何を確認するか」「何を基準に判断するか」を具体的に書けること
* 汎用内容であること（現場固有情報・機密を含まない。§4）
* 1枚15〜25行程度に収まり、短時間で読み切れること
* 初期セット（G-01〜G-20）以降の追加は、実際に現場で必要と感じた場面があってから1件ずつ登録する

### 5.3 登録しない例（両柱共通）

* AIが「重要」と言っただけで、自分ではまだ使っていない用語
* 「いつか使うかもしれない」知識
* 網羅目的で集めた関数一覧・用語集
* 具体的な確認手順・判断基準に落ちていない心構え・精神論だけのカード

## 6. 初期実装方針

初期実装では専用DB・専用管理画面は作らない。既存機能の運用で扱う。

* 既存 `notes` の `command` カテゴリ（`04-categories-and-tags.md` §1「コマンド・手順」type=command）で扱う
* 参照用タグは `dev-ref` を使う（2026-07-10 決定。旧「命名未確定」は解消。`11-open-issues.md` §15 / `04` §1.3）
* 新テーブル追加・schema変更は行わない（`03-data-model.md` の schema_version 方式に従う。当面は追加しない）
* 専用画面は作らず、既存 `/notes` の一覧・検索・絞り込み・Markdownプレビュー・コピーを利用する
* 用語・関数・ライブラリの網羅目的の大量seedは作らない。柱1は実際に手が止まったものを1件ずつ登録する運用から始める（柱2の現場理解カード初期セットG-01〜G-20はこの例外。§6.2）
* Git候補・公開の扱いは既存 `command` カテゴリの初期値（`04` §2、Git候補true）に従う。ただし現場固有情報・機密は§4のとおり登録しないため、公開しても問題ない汎用的な内容に限る

## 6.2 現場理解カードの追加と登録方式（2026-07-12）

目的を2本柱（§1）に拡張し、柱2として現場理解カード G-01〜G-20 を追加した。案Aの枠内（専用画面・新規ルート・DB変更なし、`command` カテゴリ＋タグ `dev-ref`）は変えない。

* 候補一覧・Rカードとの関係・優先度・サブタグは `25` §10 を正本とする

* 初期登録の方式：`27` のブロックをユーザーが `/notes` 作成画面へコピペして登録する。Web版DBはブラウザ内OPFSで外部スクリプトが直接触れないため（`25` R-28）、seedスクリプト方式は採らない
* 将来 `18-json-import-export.md` のJSONインポートが実装されたら、`27` の内容をJSON化して一括投入できる（`27` のブロック形式はフィールドが機械的に対応する構造にしてある）

* Git候補は `command` 初期値どおり true（Gカードは汎用内容のみ。守秘系のG-17/G-18も本文は汎用の判断基準に限定）
* 面接質問プロンプトは今回の対象外（`25` §7）
