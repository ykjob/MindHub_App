# 将来候補と不採用方針

追加日：2026-07-06

追加仕様（12〜14）で決めた採用事項は `01-decisions-and-scope.md` に記載する。

本ファイルは、将来候補と不採用の整理に使う。

## 1. 将来候補

### 1.1 Supabase同期

PCとスマホで同じDBを使いたくなった場合に検討する。

今すぐは採用しない。

検討条件。

* スマホからも編集したくなった
* PCとスマホで常に同じデータを見たくなった
* 認証、RLS、クラウドDB管理を入れてもいい段階になった

### 1.2 OpenAI API連携

現時点では採用しない。

検討条件。

* 金銭的に余裕ができた
* APIコストが下がった
* 無料または低コストで安定運用できるようになった
* アプリ内でAI整理まで完結させたくなった

当面は、プロンプトコピーしてChatGPTへ貼る運用を継続する。

### 1.3 ワンコマンド更新スクリプト

スマホ閲覧用HTML/JSON生成、git add、commit、pushをまとめるスクリプトは将来候補とする。

ただし初期は手動push。

### 1.4 カテゴリ自由追加

カテゴリ・テンプレートはDB管理にするが、初期段階では既存10カテゴリを中心に扱う。

カテゴリ自由追加は将来候補。

### 1.5 テンプレートのエクスポート・インポート

テンプレート設定をJSONで書き出し、別環境に移せる機能は将来候補。

### 1.6 mobile-inboxのGitHub連携

外出先スマホ入力をGitHub上の `mobile-inbox/` に保存する案は将来候補。

### 1.7 Phase 8ダッシュボード（旧ロードマップ）

以下のようなダッシュボードは将来候補。

* 最近更新したメモ
* プロジェクト別最新メモ
* Git候補一覧
* 未整理メモ一覧
* 次回再開メモ一覧

### 1.8 家事DB・月次作業アナウンス・自分のWebページ拡張

将来的に自分のWebページへ配置する候補。

* 家事データベース
* 生活手順
* 家の中の物の場所
* 家庭内共有メモ
* 月次でやる作業のアナウンス
* 買い物や定期作業の案内

ただし初期実装ではこれらは作らない。

### 1.9 private情報も扱えるスマホ閲覧方法

初期のスマホ閲覧ページは、公開されても困らない内容だけを扱う。

就活が終わった後、必要であればプライベート寄りの情報を扱える閲覧方法を検討する。

## 2. 不採用方針

以下は現時点では採用しない。

### 2.1 アプリ内GitHub自動push

不採用。

理由。

* GitHubトークン管理が必要
* 誤アップロードのリスクがある
* privateメモ混入時の被害が大きい

代替。

* 手動push
* 必要なら将来ワンコマンド更新スクリプト

### 2.2 Codexによる日常的なスマホ入力取り込み

不採用。

理由。

* 日常メモ入力にCodexを使うのは大げさ
* 毎回レビューや差分確認が必要になる
* スマホで書いた雑メモは、ChatGPTで構造化しGeminiへ渡す運用で十分

Codexは、実装、レビュー、コード修正、仕様整理に使う。

### 2.3 スマホからSQLite直接共有

不採用。

理由。

* PCブラウザ側SQLiteとスマホ側SQLiteは基本的に別物
* 直接共有しようとすると構成が複雑になる
* Supabaseやサーバー同期が必要になる

代替。

* スマホ閲覧用HTML/JSONを出力して見る
