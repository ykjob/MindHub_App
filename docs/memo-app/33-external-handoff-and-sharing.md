# 外部AI・共有受け渡し（Phase 16C）

追加日：2026-07-23  
文書状態：承認済み仕様／未実装  
対象Phase：Phase 16C  
対象リポジトリ：`ykjob/MindHub_App`

## 1. 文書の位置づけ

本ファイルは、MindHub内のプロンプト、さくっとメモ、現場適応の生成文を、ユーザー確認後にクリップボードまたはOS共有機能へ渡す共通仕様の正本とする。

既存仕様との分担は以下とする。

| 内容 | 正本 |
|---|---|
| ChatGPT整理プロンプト本文 | `06-chatgpt-prompt-copy.md` |
| プロンプト集の収録内容・分類 | `14-mobile-prompt-hub-and-inbox.md`および実装上のプロンプト定義 |
| さくっとメモ本文・保存・GitHub | `docs/flowdock_mvp_design.md` |
| 現場適応の質問・報告・守秘 | `21`〜`23` |
| コピー、共有確認、OS共有、Web代替、共有前保存 | 本ファイル |

本機能はOpenAI API、ChatGPT API、Slack APIを利用する直接連携ではない。

## 2. 背景

現行MindHubでは、プロンプト集、さくっとメモ本文、現場適応の質問・報告などをクリップボードへコピーできる。

コピーは、既存会話へ貼る、一部分だけ利用する、PCへ渡す等に必要であり、廃止しない。一方、スマートフォンでは次の手順が負担になる。

1. コピー
2. ChatGPT等のアプリを探して開く
3. 入力欄へ貼り付ける
4. 必要に応じて元メモも追加する

Phase 16Cでは、コピーに加えて`ChatGPTなどへ共有`を追加し、OSの共有画面からChatGPT、Gemini、Slack、メール、メモアプリ等をユーザーが選択できるようにする。

## 3. 確定した製品判断

- コピー機能を残す
- 共有ボタン名は`ChatGPTなどへ共有`
- 押下時はOSまたはブラウザの共有機能を開く
- ChatGPTだけへ直接送信しない
- 共有先はユーザーが選ぶ
- 共有前に全文を表示し、編集できる
- 使用場所ごとに共有内容の既定を決める
- 通常メモは注意文、現場適応は守秘チェック必須
- 共有用編集内容は元データを上書きしない
- 編集後の文章は、ユーザーが選べば新しい記録として保存できる
- 初期対象は次の4系統
  1. プロンプト集
  2. さくっとメモ詳細
  3. 現場適応の質問文
  4. 現場適応の進捗報告文

## 4. 目的

- コピー中心の手動操作を残しつつ、スマートフォンから外部アプリへ渡しやすくする
- 共有前に内容と守秘情報を確認できる
- プロンプトと対象本文を1つの文章にまとめられる
- MindHubから自動送信せず、最終操作をユーザーへ残す
- ChatGPTだけに依存しない汎用的な受け渡し機能にする
- WebとAndroidの双方で破綻しない

## 5. 対象外

- OpenAI API、ChatGPT API、Assistants API等の利用
- ChatGPTアプリ内部の特定会話を直接開く
- ChatGPTアプリだけを強制的に共有先指定する
- Slack API、Webhook、Botによる自動投稿
- Gemini API、Claude API等への自動送信
- 外部サービスの認証情報保存
- 共有先アプリのインストール確認
- 実際に相手へ送信されたことの保証
- 共有履歴、共有先履歴、送信済み状態
- 外部AIからの回答取り込み
- 複数メモの一括共有
- 添付ファイル、画像、音声の共有
- 自動マスキング・自動機密情報検出

## 6. 共通フロー

```text
対象画面
  ↓
［ChatGPTなどへ共有］
  ↓
外部で使う内容の確認画面
  ├─ 用途選択（さくっとメモのみ）
  ├─ 全文確認・編集
  ├─ 守秘注意／守秘チェック
  ├─ コピー
  ├─ ChatGPTなどへ共有
  ├─ 新しい記録として保存（任意）
  └─ キャンセル
  ↓
OSまたはブラウザの共有画面
  ↓
ユーザーが共有先を選択
```

MindHubは、共有画面を開いた後の送信操作を制御・保証しない。

## 7. 初期対象別仕様

### 7.1 プロンプト集

#### SHARE-PROMPT-01 共有内容

- 選択したプロンプト本文だけを共有確認画面へ渡す
- プロンプト名を確認画面上の出所情報として表示する
- 共有文章へプロンプトIDや内部管理情報を強制的に含めない

#### SHARE-PROMPT-02 操作

各プロンプトに次を表示する。

- `コピー`
- `ChatGPTなどへ共有`

既存の展開、検索、状況別絞り込み、42件の内容を変更しない。

#### SHARE-PROMPT-03 保存

編集したプロンプトを保存する場合、元のプロンプト定義を変更しない。

- `新しい記録として保存`は、既存`/notes/create`相当の記録作成へ編集文章を引き継ぐ
- ユーザーがタイトル・カテゴリ等を確認して保存する
- テンプレートDBやコード固定プロンプトを上書きしない

### 7.2 さくっとメモ詳細

#### SHARE-MEMO-01 入口

本文コピーを残し、その近くに`ChatGPTなどへ共有`を追加する。

本文が空の場合は共有不可とし、理由を文字で表示する。

#### SHARE-MEMO-02 用途選択

共有確認画面で次の5用途から選ぶ。

1. 内容を整理する
2. やることを抽出する
3. 質問内容を整理する
4. 開発依頼に整える
5. 自由に編集する

初期選択は`内容を整理する`。

用途は少数の入口であり、既存42件をすべて選択させない。

#### SHARE-MEMO-03 文章の組み立て

用途1〜4では、既存プロンプト定義を再利用し、次の構成で1つの文章を作る。

```text
【依頼内容】
＜選択用途に対応する既存プロンプト＞

【整理対象】
＜さくっとメモ本文＞
```

- 既存プロンプト本文を新しい場所へ複製して別管理しない
- 利用する既存プロンプトIDの具体的対応は、本文の意味を確認して実装時に固定マッピングする
- マッピングはコード側で明示し、名前の部分一致で自動選択しない

用途5では、メモ本文を初期表示し、ユーザーが依頼文を自由に追加・編集する。

#### SHARE-MEMO-04 元メモ非変更

共有確認画面で編集しても、`memos`の元本文を変更しない。

#### SHARE-MEMO-05 新規記録保存

- 編集文章を保存する場合は、既存`notes`の新規作成画面へ引き継ぐ
- 元のさくっとメモを上書きしない
- 保存操作を行わなければ一時文章として破棄する

### 7.3 現場適応・質問文

- `32-workplace-question-timing-rules.md`で生成した完成質問文を共有確認画面へ渡す
- プロンプトは追加せず、完成質問文だけを既定共有内容とする
- 全文編集可能
- 現場適応用守秘チェックを必須にする
- 新規記録保存は`workplace,workplace_question`、private、Git候補falseを強制する

### 7.4 現場適応・進捗報告文

- `buildReportText`等で生成した完成報告文を共有確認画面へ渡す
- プロンプトは追加せず、完成報告文だけを既定共有内容とする
- 全文編集可能
- 現場適応用守秘チェックを必須にする
- 保存する場合は既存notesを利用し、次を強制する
  - `type`: `thought`
  - `tags`: `workplace,workplace_report`
  - `visibility`: `private`
  - `is_git_candidate`: `false`
  - 一般的な日付入りタイトル

## 8. 共有確認画面

### 8.1 表示項目

- 画面タイトル：`外部で使う内容を確認`
- 出所
  - プロンプト名
  - さくっとメモ
  - 現場適応・質問
  - 現場適応・報告
- 用途選択（さくっとメモのみ）
- 編集可能な全文入力欄
- 通常注意または現場適応用守秘チェック
- コピー
- ChatGPTなどへ共有
- 新しい記録として保存
- キャンセル／戻る

### 8.2 全文入力欄

- 長文を内部スクロールできる
- 改行、日本語、記号を維持する
- 現在の編集位置がキーボードに隠れない
- 共有前に全文を確認できる
- 空文字の場合はコピー・共有・保存を無効にする

入力挙動は`31`の長文入力基準と整合させる。

### 8.3 通常注意

プロンプト集・さくっとメモでは、次の趣旨を表示する。

> 氏名、会社名、認証情報、非公開URLなど、外部サービスへ渡してはいけない情報が含まれていないか確認してください。

毎回のチェックボックスは必須にしない。

### 8.4 現場適応用守秘チェック

質問・報告では次の3項目を必須にする。

- [ ] 顧客名・会社名・個人名を含んでいない
- [ ] 認証情報・社内URL・内部システム名を含んでいない
- [ ] 会社固有の手順・判断基準・非公開情報を含んでいない

3項目すべてが確認されるまで`ChatGPTなどへ共有`を無効にする。

コピーについても外部持ち出しにつながるため、現場適応では同じチェック完了後に利用可能とする。

## 9. コピー

### SHARE-COPY-01 維持

既存コピー機能を削除・共有へ置換しない。

### SHARE-COPY-02 結果表示

- 成功：`コピーしました`
- 失敗：`コピーできませんでした`
- コピー中は二重実行を防止する
- 結果表示は一定時間後に通常状態へ戻す

### SHARE-COPY-03 コピー内容

画面に表示されている編集後全文と、クリップボード内容が一致すること。

## 10. OS・ブラウザ共有

### SHARE-OS-01 Android

- React Native／Expoで利用可能な標準共有機能を第一候補とする
- 新規依存を追加せず実現できる方法を優先する
- OSの共有先一覧を開く
- ユーザーがChatGPT等を選ぶ

### SHARE-OS-02 Web

- ブラウザ共有機能が利用可能な場合は共有画面を開く
- 利用できない場合はエラーで終了せず、次を案内する

> この環境では共有画面を開けないため、コピーして利用してください。

- コピーは引き続き利用可能にする

### SHARE-OS-03 iOS

- Phase 16Cの必須完了条件には含めない
- 標準共有機能で明らかな回帰を持ち込まない

### SHARE-OS-04 完了表示

MindHubが表示してよい表現：

- `共有画面を開きました`
- `共有を開始しました`
- `共有画面を開けませんでした`

表示してはいけない表現：

- `ChatGPTへ送信しました`
- `共有が完了しました`
- `相手に届きました`

OSやブラウザから明確なキャンセル結果を取得できない場合は、推測で成功扱いにしない。

## 11. 新しい記録として保存

### SHARE-SAVE-01 基本

- 共有用編集文章は一時データ
- 保存を選択した場合だけ、新しい`notes`レコードとして保存する
- 元のプロンプト、さくっとメモ、質問・報告の元データを上書きしない

### SHARE-SAVE-02 通常系

プロンプト集・さくっとメモからの保存は、既存の記録作成画面へ本文を引き継ぎ、ユーザーがタイトル・カテゴリ・タグ等を確認して保存する。

- 自動保存しない
- Git候補・visibilityは既存記録作成画面のルールに従う
- 引き継ぎ方法は実装判断とする

### SHARE-SAVE-03 現場適応系

質問・報告は、一般の記録作成画面へ渡して守秘既定を変更できる状態にせず、現場適応専用保存処理でprivate・Git候補falseを強制する。

### SHARE-SAVE-04 保存タイミング

`新しい記録として保存`は共有確認画面から利用できる。コピー・共有の前後を問わないが、保存は必須ではない。

## 12. 一時データ

- 共有確認画面を閉じ、保存していない編集文章は破棄してよい
- 共有履歴をDBへ残さない
- 共有先アプリ名を保存しない
- 共有日時を元メモへ記録しない
- 共有後にMindHubへ戻った場合、画面が生存している間は編集文章を維持してよい

## 13. ルート・画面構成

共有確認画面を共通ルートとして追加できる。

具体的なルート名は実装判断とするが、次を満たすこと。

- プロンプト集、さくっとメモ、質問、報告から再利用可能
- Web直アクセス時に空白・クラッシュ・機密データ表示を起こさない
- 共有対象をURLクエリへ全文で入れない
- 戻るフォールバックがある
- 一時文章を不用意にログへ出さない

## 14. データ・セキュリティ

- 共有本文をSQLiteへ自動保存しない
- URL、ログ、エラー、analyticsへ共有本文を出さない
- 認証情報を検出・送信する機能を作らない
- 外部アプリへ渡した後のデータ管理は共有先サービスの範囲であることを注意する
- 現場適応の守秘ルールは一般共有ルールより優先する
- GitHubアップロード、公開HTML、配布用出力とは別機能として扱う

## 15. エラー・例外処理

- 共有本文が空：コピー・共有・保存を無効化し理由表示
- 共有API利用不可：コピー案内
- 共有画面起動失敗：編集文章を維持し失敗表示
- 共有キャンセル：編集文章を維持し、エラー・送信済みにしない
- 保存失敗：編集文章を維持し再保存可能
- コピー失敗：共有は利用可能なまま
- 共有中の連打：多重起動防止
- 画面アンマウント後の状態更新・タイマー残留を防止

## 16. アクセシビリティ

- コピー、共有、保存、キャンセルは44相当
- 処理中、無効、結果を文字と`accessibilityState`で示す
- 守秘チェックの選択状態を読み上げ可能にする
- 共有確認全文をスクリーンリーダーで確認できる
- 最大フォントサイズでも共有・戻るへ到達できる

## 17. 受入条件

### 17.1 Gate A：プロンプト集

- [ ] 既存コピーが機能する
- [ ] 各プロンプトから共有確認へ進める
- [ ] 選択したプロンプト本文が欠けない
- [ ] 検索・絞り込み・展開・42件を壊していない

### 17.2 Gate B：さくっとメモ

- [ ] 5用途を選べる
- [ ] 既定は`内容を整理する`
- [ ] プロンプト＋本文を1つの文章として編集できる
- [ ] 編集しても元メモが変わらない
- [ ] 本文コピー、編集、削除、GitHub連携を壊していない

### 17.3 Gate C：現場適応

- [ ] 質問文・報告文を共有確認へ渡せる
- [ ] 3つの守秘チェック完了前はコピー・共有できない
- [ ] 保存時にprivate・Git候補falseが強制される
- [ ] 会社固有情報の入力を促す文言がない

### 17.4 Gate D：Android共有

- [ ] OS共有画面が開く
- [ ] ChatGPTが候補にある場合、ユーザーが選択できる
- [ ] ChatGPT未インストールでもクラッシュしない
- [ ] 共有キャンセル後にMindHubへ戻れる
- [ ] 長文・日本語・改行が欠けない
- [ ] 多重共有画面が開かない

### 17.5 Gate E：Web

- [ ] 共有対応ブラウザでは共有を開始できる
- [ ] 非対応ブラウザではコピー案内を表示する
- [ ] 360 / 390 / 412 / 480 / 768 / 1024幅で操作できる
- [ ] Web直アクセスで本文漏えい・空白画面・クラッシュがない

### 17.6 Gate F：保存

- [ ] 通常系は新しい記録作成へ引き継げる
- [ ] 現場適応系は専用安全既定で保存される
- [ ] 元データを上書きしない
- [ ] 保存しなければDBへ残らない
- [ ] 保存失敗後に再試行できる

### 17.7 Gate G：表示文言

- [ ] `ChatGPTへ送信しました`等の誤表示がない
- [ ] コピー、共有開始、保存を区別して表示する
- [ ] 共有先をMindHubが保証しないことが分かる

## 18. Claude Codeへ任せてよい判断

- React Native標準`Share`、Web Share API等の具体的利用方法
- 共通共有画面のroute名・component名・hook名
- 一時文章の安全な画面間引き継ぎ方法
- 既存プロンプトIDと5用途の固定マッピング
- Web共有可否の検出方法
- 共有起動中の状態管理
- 保存画面へのprefill方法
- 確認用テスト・ハーネス

ただし、既存プロンプトの意味を確認し、用途と合わないマッピングを行わないこと。判断理由と採用マッピングを実装報告に記載する。

任せない判断：

- コピー機能の削除
- ボタン名の変更
- API・Webhook・新規認証の追加
- 自動送信
- 共有履歴保存
- 現場適応の守秘チェック省略
- 元データ上書き
- 共有完了の断定
- 新規依存追加

## 19. 実装単位

### 16C-1 共通共有基盤

- 共有確認画面
- 編集・コピー・OS共有
- Web fallback
- 二重操作・エラー状態

### 16C-2 プロンプト集・さくっとメモ

- 共有入口
- 5用途
- プロンプト固定マッピング
- 通常注意
- 新規記録作成への引き継ぎ

### 16C-3 現場適応

- 現場適応の質問・報告の**共有接続**（16C-1の共通共有確認画面へ渡す）
- 必須守秘チェック（3項目完了前はコピー・共有できない。`23` §5.1）
- **報告文の任意保存**（現場適応専用保存処理で private・Git候補false 強制。`workplace,workplace_report`）
- Android実機・回帰確認

※**完成質問文の任意保存そのものは 16B-3**（`32` §20）が担当し、16C-3では質問・報告を共有確認画面へ接続する。同じ保存処理を16B-3と16C-3で二重実装しない（現場適応専用保存処理を共用する）。
