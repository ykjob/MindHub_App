# 現場適応モード 守秘・セキュリティとポートフォリオ説明

本ファイルは、現場適応モードの守秘・セキュリティ運用と、ポートフォリオ説明方針の正本とする。
一般的なGit除外ルールの正本は `07-export-and-git-rules.md`、公開範囲・配布区分の正本は `17-distribution-and-sharing.md`、visibility値の定義は `04-categories-and-tags.md` と `03-data-model.md`。本ファイルは現場適応モード文脈での適用方針を定義し、これらを再定義せず参照する。

## 1. 実務利用時の前提

現場適応モードは実務利用を想定するため、実際の開発現場・配属先の情報を扱う可能性がある。
そのため、守秘・セキュリティを機能の一部として設計に組み込む。

## 2. 保存してよい情報

* 一般化された作業手順・自分の理解・気づき
* 公開情報・一般的な技術知識
* 現場ルールのうち、一般化できる運用（例：「PRはレビュー後にマージ」等の抽象度）
* 架空・サンプルの現場情報（ポートフォリオ用）

## 3. 保存してはいけない情報

以下は現場適応モードに保存しない。

* 顧客名・取引先名
* 個人情報（氏名・連絡先・アカウント等）
* 認証情報（ID・パスワード・トークン・APIキー・鍵）
* 社内URL・非公開URL・内部ホスト名
* ソースコードの全文（業務コードの丸写し）
* 非公開資料の丸写し（設計書・議事録・仕様書等）

現場名・現場ルールを保存する場合も、一般化した呼称・抽象度に留める（`22` 5章）。

### 3.1 職場情報の範囲（守秘設計）

守秘対象の「職場情報」は、個人情報・顧客情報だけに限らない。以下も職場情報として扱い、現場適応モードに蓄積しない。

* 職場固有の業務ルール
* 報告経路
* 判断基準
* システム操作手順
* 例外対応
* 社内資料の内容

現場適応モードは、職場固有のルールや内部手順を個人アプリに蓄積する機能ではない。
保存対象は、作業開始・詰まり・質問・報告・終業前を支援するための、**抽象化された個人用チェック項目**に留める。

したがって、次のものは保存・Git出力・外部AI投入・公開用データ化のいずれも行わない。

* 実名・会社名・顧客名・社員名
* システム名・内部URL・具体的な画面名
* 社内マニュアル本文
* 会社固有の判断基準

現場プロファイル（`22` 5章）に現場ルールを記録する場合も、この範囲に従い、一般化・抽象化した個人用の覚書に留める。

## 4. 導線に組み込む守秘チェック

「必要なときに気をつける」だけでは守れないため、導線に注意・チェックを組み込む。

* 現場プロファイル入力画面・場面メモ入力画面に、保存禁止情報の注意を常時表示する
* 保存前・出力前に、守秘情報が含まれていないかの確認を促す
* 具体的な顧客名・非公開URLを入力しない前提を画面文言に含める（`21` 6章）

## 5. AIへ貼り付ける前の注意

質問文・報告文をChatGPT / Gemini / Claude等へ貼り付ける運用が想定されるため、貼り付け前の注意を設ける。

* AIへ渡す前に、顧客名・認証情報・社内URL・ソース全文・非公開資料が含まれていないか確認する
* 現場固有の識別情報は一般化してから貼り付ける
* この注意は整理プロンプト（`06`）・プロンプト集（`14`）の運用注意と整合させる

### 5.1 外部共有時の必須守秘チェック（2026-07-23 Phase 16C 追加）

Phase 16C（`33-external-handoff-and-sharing.md`）で現場適応の質問・報告を外部アプリへ共有・コピーする場合、次の3項目を**必須チェック**とする（本章3〜3.1の保存禁止情報を弱めるものではなく、外部持ち出し時に上乗せする）。

* [ ] 顧客名・会社名・個人名を含んでいない
* [ ] 認証情報・社内URL・内部システム名を含んでいない
* [ ] 会社固有の手順・判断基準・非公開情報を含んでいない

* 3項目すべてが確認されるまで、`ChatGPTなどへ共有`（OS共有）も**コピー**も無効にする（現場適応系のみ。通常系＝プロンプト・さくっとメモは注意文で、毎回のチェックボックスは必須にしない。`33` §8.3〜8.4）
* Phase 16の質問文・報告文を**新規保存する操作**では、`visibility=private`・`is_git_candidate=false` を**強制し、保存画面で変更させない**（現場適応専用保存処理。`22` §8.1・`04` §1.2.1・`33` §7.3〜7.4）
* **OS共有も外部送信に該当する**。コピー同様、外部へ持ち出す操作として扱う
* MindHubは共有画面を開くところまでを扱い、**共有先アプリ・共有後のデータ管理はMindHubの範囲外**である（`33` §14）。「送信しました」「共有完了」とは表示しない（`33` §10）

Phase 16の質問・報告保存の強制（変更させない）は、§6.1の一般の現場適応メモに対する「既定は安全側・ユーザーが個別に変更することは妨げない」ルールとは**区別**する。両者の関係は §6.1 に整理する。

## 6. 既定値による安全側設計

* visibility の既定は `private`（`03` / `04` 参照）
* is_git_candidate の既定は `false`（`04` / `07` 参照）
* 現場適応モードで生成・保存する情報は、既定で「外に出さない」側に倒す

### 6.1 カテゴリ初期値に対する守秘優先の上書き（要）

現場適応モードでは、既存type（worklog / research 等）を再利用する場合がある。これらのカテゴリは `04-categories-and-tags.md` §2 でGit候補初期値が `true` のものがあるが、**現場適応モードではカテゴリのGit候補初期値に依らず、守秘優先で上書きする**。

* 場面メモ・現場情報・現場由来の作業支援メモは、既定で `is_git_candidate = false` かつ `visibility = private` とする
* この上書きは `04` §2 のカテゴリ初期値より優先される（優先順位：現場適応モードの守秘ルール ＞ カテゴリ初期値）
* ユーザーが個別に変更することは妨げないが、既定は常に安全側（false / private）から始める

この上書きルールは `22-workplace-adaptation-data-and-integration.md` §4 のデータ表現と、`07-export-and-git-rules.md` §13 のGit除外に整合させる。

#### 6.1.1 一般の現場適応メモとPhase 16の質問・報告保存の区別（2026-07-23）

* §6.1（上記）は、現場適応モードの**一般の場面メモ・現場情報**に対する既定ルールであり、既定は安全側（false / private）だが、ユーザーが個別に変更することは妨げない
* 一方、Phase 16（`32`／`33`）の**質問文・報告文を新規保存する操作**は、現場適応専用保存処理で `visibility=private`・`is_git_candidate=false` を**強制し、その保存画面では変更させない**（§5.1）
* この2つは競合ではなく、対象と操作が異なる（一般メモの既定 vs Phase 16の新規保存の強制）
* **保存後に、一般の記録編集画面から private・Git候補false を変更可能にするかどうかは今回決定しない**。永続的な変更禁止や専用ロック機能を勝手に採用しない。この「保存後の再編集時の扱い」は `11-open-issues.md` に未決定事項として記録する

## 7. 出力・配布物への非混入

* 現場情報を公開GitHub Pages（スマホ閲覧用HTML／`13-mobile-view-export.md`）に出さない
* 現場情報を配布用リポジトリに含めない（`17` の三区分：配布用は汎用データのみ）
* 現場情報を家族用の閲覧対象に含めない（`17` 家族用は家庭内情報のみ）
* これらは `07` のGit除外・`17` の配布区分・`13` の公開判定（mobileViewPolicy）と整合させる

## 8. ポートフォリオでの扱い

### 8.1 架空データを使う

* デモ・公開・面接提示の際は、実在現場ではなく架空データ（サンプル現場・サンプルタスク）を使う
* スクリーンショット・録画にも実在の顧客名・非公開情報を映さない

### 8.2 採用担当者への説明方針

複数現場を経験した採用担当者・現場担当者が見る前提で、以下を説明できる構成にする。

* 機能の羅列ではなく、毎日必ず発生する5場面（`21`）に導線を組み込んだ「場面駆動」であること
* 未経験者の抱え込み防止（詰まり記録）と、習慣化（作業開始・終業前）を狙った設計であること
* 守秘・セキュリティを機能として組み込んでいること（本ファイル3〜7章）
* 既存アプリ（notes等）と整合し、破壊的変更を避けていること（`22`）

### 8.3 現場経験者が見て不安を感じにくい説明の型

* 「何を保存しないか」を先に示す
* 既定値が安全側（private / Git候補false）であることを示す
* 公開・配布・家族共有には現場情報が出ない設計であることを示す
* 実データではなく架空データで運用・提示することを明示する
