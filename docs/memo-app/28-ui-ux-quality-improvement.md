# UI・UX品質改善（Phase 15）

追加日：2026-07-13

本ファイルは、Phase 15（UI・UX品質改善）における**画面別改善と情報設計の正本**とする。

正本の分担は以下のとおり。

* 本ファイル（`28`）：Phase 15の目的・UX問題一覧・情報設計・画面別改善方針・変更範囲・追跡表
* `29-ui-design-system.md`（後続作成予定・未作成）：デザイントークン（色・余白・文字・角丸）、共通UIコンポーネント詳細、ヘッダー実装仕様、状態表示（LoadState）の設計、アクセシビリティ属性の詳細
* `30-ui-validation-checklist.md`（後続作成予定・未作成）：画面幅別チェック項目、状態別確認、既存機能の回帰確認、Android実機確認手順

ロードマップ上の位置づけは `09-roadmap.md`（Phase 15。追記は後続の文書整備で行う）、細分化タスクは `10-tasks.md`、未確定事項は `11-open-issues.md` で管理する。

## 1. 目的

Phase 15は機能追加のフェーズではない。現在の機能を、理解しやすく、操作しやすく、統一感があり、説明しやすい形へ改善するフェーズである。

最終的に、画面だけを見た人が次のように理解できる状態を目指す。

> MindHubは、考えを素早く記録し、あとで整理し、仕事中の詰まり、質問、報告、翌日の再開までを支援するアプリである。

具体的には以下を実現する。

* MindHub全体の役割をホームで理解できる
* FlowDock由来の軽量メモ、メモ管理、現場適応モード、プロンプト集の違いが分かる
* 使用頻度と重要度が視覚的に表現される
* WebとAndroidで同じ設計意図が保たれる
* 主要画面のヘッダーと共通部品が統一される
* エラー、空状態、読み込み中、操作結果が区別される
* スマホ幅での情報密度が下がる
* 現場適応モードの一日ループ（開始→作業中→終了→翌朝再開）が伝わる
* プロンプトを利用状況から探せる
* ポートフォリオとして独自性を説明しやすくなる

## 2. 背景

### 2.1 なぜPhase 15を独立フェーズにするか

Phase 0〜14までで、メモ管理（notes）・プロンプト集・現場適応モード・Android APK版の各機能は実用段階に到達した。一方で、UIは各Phaseで画面単位に追加されてきたため、次の状態にある。

* デザイン値（色・余白・文字サイズ等）が各画面のStyleSheetに個別定義されている
* ヘッダー方式が画面ごとに違う（ネイティブStackヘッダーと画面内独自ヘッダーの混在）
* ホームが「memos一覧＋機能ボタン列」のままで、アプリ全体の入口として機能していない
* 名称（MindHub / FlowDock）が画面上で混在している

UIだけを場当たり的に変更すると、仕様書と実装がずれたり、既存の変更禁止範囲（DB・保存処理・守秘既定）を誤って触る危険がある。そのため、正式仕様（本ファイル）とタスク構造を先に整備し、1タスク1目的で進める。

### 2.2 4視点レビューの指摘

以前実施した4視点レビュー（プロダクト／技術／UI・UX／品質・セキュリティ）で、UI・UX面の問題16件（UX-01〜UX-16）が指摘された。Phase 15はこの16件を取りこぼさず扱い、最後に再評価（改善済み／部分改善／未改善の判定）まで行う。

## 3. 現在評価されている問題（概観）

問題は大きく5系統に分かれる。

1. **情報設計**：ホームが入口に見えない、機能の違い・優先順位が伝わらない、名称混在（UX-01〜04、16）
2. **共通UI基盤**：ヘッダー混在、デザイン値分散、視覚階層の弱さ（UX-05〜07）
3. **画面別の問題**：メモ管理のフィルター過多・チップの意味不統一、現場適応のフォーム一覧感・保存とコピーの区別、プロンプトの状況別検索（UX-08〜12）
4. **状態表示**：エラーと空状態の区別なし（UX-13）
5. **操作性・密度**：タッチ領域・アクセシビリティ不統一、スマホ幅の情報密度（UX-14〜15）

## 5. 改善の基本原則

1. **機能追加ではなく再構成**：新機能・新DBテーブル・新保存処理を増やさない
2. **既存機能を壊さない**：`/memo` 系（FlowDock由来）・`/notes`・`/workplace`・`/prompts`・`/settings` の既存ルートと保存ロジックは不変
3. **1タスク1目的**：複数画面の同時変更をしない。ついで修正をしない。別問題を見つけたら新タスク候補として記録する
4. **共通UI基盤を先に、画面刷新を後に**：デザイントークンと共通部品（正本は後続の `29`）を先に整備し、各画面はそれを使って順に改善する
5. **意味の固定、多色化の回避**：色は増やすためでなく意味（主要操作／保存・完了／注意／エラー／AI）を固定するために使う。過剰な装飾・アニメーションは入れない
6. **WebとAndroidの両方で確認**：各段階でWeb（PC幅・スマホ幅）を確認し、区切りでAndroid APK実機確認を行う（確認手順の正本は後続の `30`）
7. **色だけで状態を示さない**：文字・ラベル・枠・アイコンを併用する
8. **未確定事項を勝手に確定しない**：判断が必要な項目は `11-open-issues.md` に記録し、ユーザー判断を待つ

## 6. MindHub全体の情報設計

### 6.1 機能系統と役割（名称正本は `01` §1.6）

| 系統 | 役割 | ユーザーへの見せ方 |
|---|---|---|
| `/memo` 系（FlowDock由来） | スマホで素早く記録する軽量メモの入口 | 「すぐにメモする」機能として提示 |
| `/notes` 系 | タイトル・カテゴリ・タグ・Git候補で整理するメモ管理 | 「記録を整理する」機能として提示 |
| `/workplace` 系 | 現場適応モード。仕事の5場面（開始・詰まり・質問・報告・終業前）＋翌朝再開 | 「仕事を整理する」機能として提示。MindHubの独自価値の中心 |
| `/prompts` 系 | プロンプト集。AI（ChatGPT / Gemini / Claude Code / Codex）活用の型 | 「AIの型を使う」機能として提示 |
| `/settings` | GitHub連携設定 | 補助操作。主要機能と同列に並べない |

### 6.2 優先順位

* **最重要**：軽量メモ（すぐ記録）、現場適応モード（毎日の仕事場面）
* **主要**：メモ管理、プロンプト集
* **補助**：設定

この優先順位を、ホームのカードの大きさ・位置・見た目の差で表現する（UX-03、07対応）。

### 6.3 導線の原則

* ホームはMindHub全体の入口とし、各機能へ1タップで到達できる
* 各機能内の既存導線（一覧→詳細→編集等）は変更しない
* 機能間の新しい横断導線（例：詰まり記録→プロンプト集）はPhase 15では追加しない（将来候補）

## 7. ホーム画面の改善方針

対象：`app/index.tsx`（画面内容の再構成。ルート `/` は不変）。

### 7.1 役割の再定義

ホームを「memos一覧画面」から「MindHub全体の入口」へ再構成する。既存の `08-ui-flow.md` §1（メイン画面＝メモ状況の把握）と `21` §1（既存ホームへの変更は入口ボタン追加のみ）は、この方針で置き換わる（両文書への反映は後続の文書整備で行う）。

この役割はユーザー確定済み（2026-07-14 IA判断）。ホームは軽量メモ一覧を主目的とする画面ではなく、MindHub全体の主要機能へ進む入口として再構成する。バッチ1でAppHeaderを適用した現在のホーム（AppHeader＋既存の「メモ一覧」ヘッダー・4ボタン・memos一覧）は暫定状態であり、バッチ2（ホーム再構成）完了までは完成扱いにしない。

### 7.2 構成要素（2026-07-14 IA判断で確定）

* MindHubタイトルと1行の説明（アプリ全体の目的が分かる短文）
* 主要機能カード：次の4件とし、表示順も次の順を基本とする。各カードにはカード名だけでなく短い役割説明を付ける。§6.2の優先順位を見た目に反映する
  1. **すぐメモする**（役割：思いついた内容を素早く残す。遷移先：`/memo/create`）
  2. **仕事を整理する**（役割：仕事中の5つの場面を整理する。遷移先：`/workplace`。ホームカード上の入口ラベルのみで、正式名称「現場適応モード」は変更しない＝§11）
  3. **記録を整理する**（役割：保存した記録を確認・分類する。遷移先：`/notes`）
  4. **AIの型を使う**（役割：目的別のプロンプトを選んでコピーする。遷移先：`/prompts`。バッチ2では既存 `/prompts` へ遷移させるだけとし、プロンプト本文・分類・生成処理・プロンプト集画面は変更しない）
* 最近の軽量メモ（取得元はmemosのみ。notesとの統合取得はしない）：既存のmemos取得処理と既存の最新順を利用し、初期表示は最大3件。項目選択時は既存の軽量メモ詳細へ遷移する。3件を超える場合は「すべて表示」でホーム内に展開し、「3件に戻す」等の補助操作で初期状態へ戻せる（3件以下では展開操作を表示しない）。専用の軽量メモ一覧ルートは追加しない＝別ルートへ移動する「すべて見る」ではなくホーム内展開とする（実際の文言は実装時に既存UIとの整合を確認）。DBスキーマ・並び順の定義は変更しない
* 設定：主要機能カードには含めず、主要カード群と視覚的に分離した補助導線とする。AppHeader右側に置くか、ホーム内の補助位置に置くかは固定せず、バッチ2の既存実装調査後に共有AppHeaderを不必要に変更しない最小構成を選ぶ
* FAB：既存FABは、スクロール位置にかかわらず軽量メモ作成へ進める即時入力導線として維持する。「すぐメモする」カードと役割が一部重複するが、バッチ2では削除しない（カード＝機能の発見性、FAB＝即時操作性。実機確認後に評価する）

### 7.3 維持するもの

* メモ作成FAB相当の「すぐ記録できる」導線（軽量メモへの最短経路を殺さない）
* 既存の遷移先（`/memo/*`・`/notes`・`/workplace`・`/prompts`・`/settings`）
* GitHub連携バッジ等、memos一覧の既存表示要素は「最近のメモ」内または遷移先一覧で維持する

### 7.4 スマホ幅

360 / 390 / 412 / 480 / 768 / 1024以上で崩れないこと（チェック項目の正本は `30`）。2×2グリッド（e12aed6）のAndroid実機基準確認は完了し（`30` §12.2）、IA判断も確定済み（2026-07-14）のため、ホーム再構成（バッチ2）に着手できる。

## 8. メモ管理画面の改善方針

対象：`app/notes/index.tsx` を中心とする一覧まわり。保存処理・検索クエリは不変。

### 8.1 フィルターの収納（UX-08）

* 常時表示：検索、カテゴリ、絞り込みボタン、有効中の条件表示
* 収納（開閉式等に格納）：プロジェクト、タグ、Git候補、アーカイブ、並び順
* 収納UIの具体方式（折りたたみ／モーダル等）は実装タスクで確定する

### 8.2 チップの役割分離（UX-09）

フィルター選択・オン/オフ切り替え・並び順・実行ボタンを見た目と操作で区別する。チップの具体仕様は後続の `29` を正本とする。

### 8.3 一覧の状態区別（UX-13）

読み込み中／データなし／条件一致なし／読み込み失敗を区別して表示する。特に「検索条件に一致しない」と「メモが1件もない」と「読み込みに失敗した」を同じ空表示にしない。状態表示の実装設計は後続の `29` を正本とする。

### 8.4 スマホ幅の情報密度（UX-15）

フィルターが一覧を圧迫しないことをスマホ幅で確認する。`/notes` ヘッダーの2段化要否はAndroid実機確認の結果を見て判断する（既存の判断待ち事項を引き継ぐ）。

## 9. 現場適応モード入口の改善方針

対象：`app/workplace/index.tsx`（入口のみ）。5場面の各画面・`WorkplaceSceneForm`・保存処理・守秘既定（`23` §6.1）は不変。

### 9.1 一日の流れとして表示（UX-10）

5場面カードの縦並びを、一日の流れが分かる3区分に再構成する。

* 開始：作業開始（＋前回の再開メモからの引き継ぎ）
* 作業中：詰まり記録、質問文作成、進捗報告作成
* 終了：終業前メモ

### 9.2 動作種別の表示（UX-11)

各場面カードに、その場面の出力動作を明示する。

* コピーのみ：作業開始、詰まり記録、質問文作成、進捗報告作成
* コピーと保存：終業前メモ（visibility=private / Git候補false固定。保存仕様は不変）

### 9.3 前回メモとの統合（既存機能の見せ方改善）

「前回の再開メモ」カードと「作業開始」を、翌朝再開のループが分かる形で自然につなげる。`?fromRestart=1` の既存挙動は変更しない。

### 9.4 独自価値の説明（UX-16）

入口の説明文で「作業の状況を整理し、質問、報告、翌日の再開につなげる」ことが分かるようにする。守秘注意文の常時表示は維持する。

## 10. プロンプト集の改善方針

対象：`app/prompts/index.tsx`。プロンプトデータ（`promptHub.ts`）の本文・件数は不変。

### 10.1 状況別入口の追加（UX-12）

名称・分類検索に加えて、利用状況から選ぶ入口を追加する。分類候補（確定は実装前判断）：

* 問題を整理する
* エラーを相談する
* 作業を始める
* 報告文を作る
* 引き継ぎを作る
* 仕様を確認する

### 10.2 既存検索の維持

状況別入口を追加しても、現在の検索・セクション別一覧・展開・コピーは残す。既存導線の削除・置き換えはしない。

### 10.3 状態表示の共通化

コピー成功・失敗の表示を共通部品（後続の `29`）へ寄せる。現在の「コピーしました／コピー失敗」の挙動水準は維持する。

## 11. 名称と機能系統の整理（UX-02、04）

名称方針の正本は `01-decisions-and-scope.md` §1.6（MindHub＝アプリ全体名、FlowDock＝軽量メモ機能の説明名）。Phase 15では、この方針を画面表示へ反映する。

* 画面上の全体名は MindHub に寄せる（対象：画面内タイトル・説明文などの表示文言）
* FlowDock は軽量メモ機能の説明名として、説明文の中で残してよい（例：「クイックメモ（FlowDock）」等。表記は実装前判断）
* 軽量メモの表示名候補は「クイックメモ」（確定は実装前判断）
* 現場適応モードの画面上の短い表示名は、ホームカード上のみ「仕事を整理する」で確定（2026-07-14 IA判断）。正式名称の変更ではなく、ホーム上の入口ラベルだけの変更。正式説明名は「現場適応モード」のまま
* **変更しない**：ルート名（`/memo` 等）、DBテーブル名、package.json、実装上の識別子

## 12. 旧ダッシュボード構想との境界

Phase 15のホーム改善は、旧Phase 8のダッシュボード構想（`15-future-and-rejected-policies.md` §1.7、`09-roadmap.md` Phase 13）の全面実装ではない。両者を明確に分離する。

### 12.1 Phase 15の軽量ホームに含めるもの

* 主要機能への分かりやすい導線
* 機能の優先順位の視覚表現
* 短い説明文
* 最近のメモ最大3件程度
* 設定を補助操作へ移す方針

### 12.2 Phase 15に含めないもの（将来候補のまま）

* 集計グラフ・分析機能
* プロジェクト別統計・カテゴリ別統計
* Git候補一覧・未整理メモ一覧・次回再開メモ一覧などの複合パネル
* 複雑なダッシュボードレイアウト
* 新しいDBクエリ基盤
* 新しい保存処理

「最近のメモ3件」は既存の一覧取得の範囲で実現し、新しい集計クエリを作らない。将来ダッシュボードを実装する場合は別Phaseとして改めて判断する。
