## 2. 背景

現行の現場適応モードでは、詰まり記録で状況・試したこと・確認したいことを整理し、質問文作成で相手が答えやすい文章を生成できる。

一方、未経験者は次の判断に迷いやすい。

- まだ自分で調べるべきか
- 何分程度で質問へ切り替えるべきか
- 影響が大きい作業を自分の判断で進めてよいか
- 急ぎの確認と、通常の質問をどう分けるか
- 質問前に最低限何を確認すればよいか

本機能は、正解を自動決定するのではなく、質問タイミングを安全側に整理し、質問文作成へつなげるための補助機能とする。

## 3. 確定した製品判断

- 通常の調査目安は20分
- 完全に作業が止まっている場合の目安は10分
- 本番、顧客、個人情報、認証、権限、削除、セキュリティ等の危険条件では時間を待たず確認する
- 現場で正式な質問ルールが定められている場合は、そのルールを優先する
- 初期実装では10分・20分をユーザー設定できる画面を作らない
- 判定途中の入力と判定結果はDBへ保存しない
- 完成した質問文だけ、ユーザーが選んだ場合に保存できる
- 導線は「詰まり記録→質問タイミング確認→質問文作成」
- `判定せず質問文を作る`導線を用意し、判定を強制しない

## 4. 目的

- 質問を遅らせるべきでない条件を先に見つける
- 20分・10分の目安を、絶対ルールではなく安全な切り替え目安として使う
- 「なぜ今質問するのか」「なぜもう少し調べるのか」を表示する
- 質問時に必要な情報を、質問文作成画面へ引き継ぐ
- 質問をためらって抱え込むことを減らす
- 相手へ丸投げに見えにくい質問文を作る

## 5. 対象範囲

- 詰まり記録からの質問タイミング確認導線（現場適応入口から開く独立機能は追加しない）
- 質問タイミング確認画面
- 3段階の判定結果
- 質問文作成画面への一時引き継ぎ
- 完成質問文のコピー、外部共有、任意保存
- 一般化された守秘確認

## 6. 対象外

- AIによる自由文の自動リスク判定
- 会社固有のSLAや質問ルールの自動取得
- 質問先、回答期限、回答状況の監視
- Slackやメールへの自動送信
- タイマー通知、プッシュ通知
- メンバー・上司・チーム管理
- 判定履歴、質問履歴の専用テーブル
- 職場別の10分・20分設定画面
- 質問した回数や回答速度の評価
- 人事評価、勤務評価への利用

## 7. 基本原則

### 7.1 現場ルール優先

画面上に次の趣旨を表示する。

> 現場で質問・確認の時間や手順が決められている場合は、そのルールを優先してください。

ユーザーが「現場ルールあり」を選んだ場合、MindHub独自の10分・20分より現場ルールを優先する旨を結果へ表示する。現場ルール本文は保存しない。

### 7.2 危険条件優先

危険・高影響条件が1つでも該当する場合、経過時間や調査状況にかかわらず「すぐ確認する」を最優先する。

### 7.3 時間は目安

10分・20分は質問を禁止する時間ではない。短時間でも危険、権限外、期限影響がある場合は質問する。

### 7.4 ユーザーが最終判断する

結果は推奨であり、ユーザーは次のいずれも選べる。

- 質問文を作る
- もう少し調査する
- 判定を無視して質問文を作る
- 現場適応モードへ戻る

## 8. 画面フロー

```text
詰まり記録
  ↓
［質問タイミングを確認］
  ↓
質問タイミング確認
  ├─ すぐ確認する
  ├─ 早めに質問する
  └─ もう少し調査する
       ↓
［質問文を作る］または［調査へ戻る］
       ↓
質問文作成
       ↓
整理済み質問文
  ├─ コピー
  ├─ ChatGPTなどへ共有
  └─ 記録として保存（任意）
```

質問タイミング確認画面には、`判定せず質問文を作る`導線を置く。

## 9. 入力項目

入力負担を増やしすぎないため、初期実装では以下の6群に限定する。

### QT-INPUT-01 現場ルール

- 現場で質問タイミングや確認手順が決められている
  - はい
  - いいえ／分からない

「はい」の場合もルール本文は入力・保存させない。画面上で「現場ルールを優先」と案内する。

### QT-INPUT-02 危険・高影響条件

複数選択可。

- 本番環境・公開環境へ影響する
- 顧客・利用者へ影響する
- 個人情報・機密情報を扱う
- ID、パスワード、トークン、APIキー、権限を扱う
- データ削除、上書き、移行など元に戻しにくい操作
- セキュリティ、障害、事故につながる可能性
- 外部への送信、公開、リリースを伴う
- 自分の権限・担当範囲で判断してよいか分からない
- 該当なし

具体的な会社名、システム名、URL、認証情報は入力しない。

### QT-INPUT-03 作業停止状態

- 完全に止まっていて、次に安全に試せることがない
- 一部は進められる
- まだ調査を始めたばかり

### QT-INPUT-04 経過時間

- 10分未満
- 10分以上20分未満
- 20分以上

正確な時刻入力、タイマー、通知は初期実装に含めない。

### QT-INPUT-05 期限・周囲への影響

複数選択可。

- 今日中・直近の期限に影響する
- 他の人の作業を止める
- 手戻りが大きくなる可能性がある
- 仕様・要件の判断が必要
- 自分の裁量を超えている
- 該当なし

### QT-INPUT-06 最低限の確認

質問前に確認できる一般項目。

- 指示・チケット・仕様を読み直した
- マニュアル・README・過去記録を確認した
- エラー表示・ログ・再現条件を確認した
- 影響の小さい範囲で試せることを試した
- 何を聞きたいかを一文で言える

危険条件がある場合は、これらが未完了でも質問を遅らせない。

## 10. 判定結果

判定は次の3種類とする。

| 結果 | 意味 |
|---|---|
| すぐ確認する | 時間を待たず、作業を進める前に確認する |
| 早めに質問する | 現時点で質問へ切り替えることを推奨する |
| もう少し調査する | 影響の小さい範囲で確認を続けてよい |

### 10.1 優先順位

```text
すぐ確認する ＞ 早めに質問する ＞ もう少し調査する
```

複数条件が該当する場合は、最も緊急度の高い結果を採用する。

## 11. 判定ルール

### QT-RULE-01 すぐ確認する

以下のいずれかに該当した場合は「すぐ確認する」。

- QT-INPUT-02の危険・高影響条件が1つ以上
- 操作を続けると元へ戻せない可能性がある
- 権限・担当範囲を超える判断が必要
- 事故、漏えい、障害、顧客影響を防ぐため確認が必要

※「現場ルールで即時確認が求められている」かどうかは、QT-INPUT-01が「現場ルールあり／なし」しか取得せず、ルール本文や即時確認条件を取得しないため、MindHubは自動判定しない。現場ルールありの場合は QT-RULE-04 に従い「現場ルールを優先」と注意表示するに留め、ルールの具体的内容を推測しない（ルール本文の入力・保存も行わない）。

結果理由の例：

- 「本番・顧客への影響があるため、時間を待たず確認してください」
- 「元に戻しにくい操作のため、実行前の確認を優先してください」
- 「権限・担当範囲の判断が必要です」

### QT-RULE-02 早めに質問する

危険条件がなくても、以下のいずれかに該当した場合は「早めに質問する」。

- 完全に作業が止まった状態で10分以上経過
- 通常の調査を20分以上続けている
- 期限、他者の作業、手戻りへ影響する
- 仕様・要件の判断が必要
- 自分の裁量を超えている
- 最低限の確認を行ったが次の安全な調査手段がない

結果理由の例：

- 「完全に止まった状態が10分以上続いています」
- 「通常の調査目安20分を超えています」
- 「他の作業や期限への影響があるため、早めの共有が適切です」

### QT-RULE-03 もう少し調査する

次をすべて満たす場合に限り「もう少し調査する」。

- 危険・高影響条件がない
- 完全停止ではない、または完全停止10分未満
- 通常20分未満
- 期限・他者・手戻りへの大きな影響がない
- まだ確認していない一般的な資料または安全な試行がある

結果には、未確認項目から最大3件の次の行動を表示する。

例：

- 「指示・仕様をもう一度確認する」
- 「エラーと再現条件を整理する」
- 「あと10分調べる」のような強制カウントダウンは表示しない

### QT-RULE-04 現場ルールあり

- 現場ルールがある場合、結果の最上部に「現場ルールを優先」と表示する
- MindHubの判定が現場ルールと異なる可能性があることを明示する
- 現場ルール本文は記録しない

## 12. 結果画面

結果画面には以下を表示する。

1. 判定結果
2. 判定理由（最大3件）
3. 次に行うこと
4. 注意文
5. 操作ボタン

### 12.1 操作ボタン

結果が「すぐ確認する」「早めに質問する」の場合：

- `質問文を作る`（主要操作）
- `詰まり記録へ戻る`
- `現場適応へ戻る`

結果が「もう少し調査する」の場合：

- `調査内容を見直す`（詰まり記録へ戻る）
- `質問文を作る`
- `現場適応へ戻る`

判定結果によって質問文作成を禁止しない。

## 13. 質問文作成への引き継ぎ

判定画面の入力値と結果はDBへ保存せず、画面遷移中の一時情報として質問文作成へ引き継ぐ。

引き継ぎ候補：

- 判定結果
- 判定理由
- 急ぎ度の候補
- 詰まり記録の状況
- 試したこと
- 確認したいこと
- エラー内容

質問文作成画面では、引き継いだ内容を編集できること。

### 13.1 質問文への反映

- 「すぐ確認する」：急ぎ度へ「作業を進める前に確認したい」を候補表示
- 「早めに質問する」：急ぎ度へ「早めに確認したい」を候補表示
- 「もう少し調査する」から質問文作成へ進んだ場合：急ぎ度を自動決定しない

判定理由をそのまま相手へ送る必要はない。質問文へ含めるかはユーザーが編集する。

## 14. 完成質問文の保存

### QT-SAVE-01 保存対象

- 判定途中の入力、判定結果、チェック状態は保存しない
- `buildQuestionText`等で生成され、ユーザーが確認した完成質問文だけ保存対象にできる

### QT-SAVE-02 保存操作

- 質問文の出力後に`記録として保存`を表示する
- 保存は任意であり、コピー・共有に必須ではない
- 保存後もコピー・共有を利用できる

### QT-SAVE-03 保存先と既定

既存`notes`を再利用し、破壊的変更を行わない。

- `type`: `thought`
- `tags`: `workplace,workplace_question`
- `visibility`: `private`
- `is_git_candidate`: `false`
- `source`: 既存手動保存の値
- タイトル：日付を含む一般的な名称（例：`質問メモ YYYY-MM-DD`）

守秘既定はカテゴリ初期値より優先する。

### QT-SAVE-04 二重保存

- 保存中の二重実行を防止する
- 保存成功後は同じ出力を重複保存できない状態にする
- 出力を編集・再生成した場合は、新しい内容として保存可能に戻してよい

## 15. 守秘・セキュリティ

- 顧客名、会社名、氏名、内部URL、システム名、認証情報、社内資料本文、会社固有の判断基準を入力しない
- 危険条件は一般化された選択肢のみとし、具体的な対象名を入力させない
- 外部共有前は`33`で定義する現場適応用チェックを必須にする
- 保存時はprivate・Git候補外を強制する
- 公開HTML、家族用、配布用、GitHub出力へ含めない

## 16. エラー・例外処理

- 判定入力が不足していてもクラッシュしない
- 危険条件が未選択でも、未選択を安全と断定しない。選択を促す
- 保存失敗時も完成質問文を維持する
- 質問画面への引き継ぎに失敗した場合は、空フォームを表示し、詰まり記録へ戻れるようにする
- 戻る操作で判定途中データが失われることは許容するが、その旨を仕様上の一時データとして扱う

## 17. アクセシビリティ

- 選択肢は色だけで状態を示さない
- チェック・ラジオ相当の状態を`accessibilityState`で示す
- 判定結果は見出しと文章で示す
- 主要操作は44相当
- 最大フォントサイズでも結果と操作へ到達できる

## 18. 受入条件

### 18.1 Gate A：判定ロジック単体

- [ ] 危険条件1件で「すぐ確認する」
- [ ] 完全停止10分以上で「早めに質問する」
- [ ] 通常20分以上で「早めに質問する」
- [ ] 10分未満・低影響・未確認項目ありで「もう少し調査する」
- [ ] 複数条件では最も緊急な結果になる
- [ ] 現場ルールありで優先注意を表示する

### 18.2 Gate B：画面フロー

- [ ] 詰まり記録から判定へ進める
- [ ] 判定をスキップして質問文作成へ進める
- [ ] 3結果それぞれから質問文作成へ進める
- [ ] 詰まり内容・試したことを質問フォームへ引き継げる
- [ ] 戻る操作で行き止まりにならない

### 18.3 Gate C：保存

- [ ] 判定途中データがDBへ保存されない
- [ ] 完成質問文だけ任意保存できる
- [ ] `workplace,workplace_question`、private、Git候補falseで保存される
- [ ] 保存失敗時に質問文が残る
- [ ] 二重保存されない

### 18.4 Gate D：共有・守秘

- [ ] 3つの守秘チェック完了前はコピー・共有できない
- [ ] 3つの守秘チェック完了後にコピーできる
- [ ] 3つの守秘チェック完了後に `ChatGPTなどへ共有`へ進める
- [ ] 共有キャンセルで質問文が失われない
- [ ] 「送信完了」と誤表示しない

Gate Dの共有・守秘確認は、共通共有基盤（16C-1）と現場適応共有接続（16C-3）の実装後に**統合時に通過させる連携Gate**である。16Bの判定ロジック（16B-1）・判定画面と導線（16B-2）・完成質問文の任意保存（16B-3）は、16Cの共有基盤がなくても実装・確認でき、Gate D（共有）未通過だけを理由に不合格にしない（`33` §17 Gate A〜Gと統合して確認する）。

### 18.5 Gate E：回帰

- [ ] 作業開始、詰まり、報告、終業前、翌朝再開を壊していない
- [ ] 終業前メモの既存保存・再開導線を壊していない
- [ ] WebとAndroidで操作不能にならない
- [ ] DBスキーマを変更していない

## 19. Claude Codeへ任せてよい判断

- 判定関数の配置、型名、純粋関数化
- 一時引き継ぎにroute params、context、画面内state等のどれを使うか
- 選択UIの具体的な共通部品
- 質問保存処理を既存`saveWorkplaceNote`相当へ統合する方法
- テストケースの実装方法

- 10分・20分の変更
- 判定結果の追加・削除
- 危険条件の弱体化
- 判定履歴の保存
- AI自動判定
- 現場ルール本文の保存
- private・Git候補falseの解除
- 判定の強制化

## 20. 実装単位

### 16B-1 判定ロジック

- 入力型
- 3段階判定
- 理由生成
- 単体確認

### 16B-2 判定画面と導線

- 詰まり→判定→質問
- スキップ
- 一時引き継ぎ
- Web幅別確認

### 16B-3 完成質問文の任意保存

- 完成質問文の任意保存のみ（既存notes・`workplace,workplace_question`・private・Git候補false。`14. 完成質問文の保存`）
- 保存の二重実行防止・保存失敗時の質問文維持
- 共有確認画面（`ChatGPTなどへ共有`）への接続・必須守秘チェック・報告文の共有/保存は **16C-3** が担当する（`33` §19）。16B-3では共有処理を実装しない
- Android実機・回帰確認は保存部分について行う
